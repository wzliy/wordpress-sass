# 跨境电商建站 SaaS 系统架构文档

## 1. 文档目标

本文档用于重新定义当前系统的目标架构。新系统不再以“WordPress 多站点接入 + 内容发布”为中心，而是重构为面向跨境电商卖家的建站 SaaS 平台。

核心目标：
- 一键建站
- 斗篷能力
- ERP 能力
- 支付能力
- 后台可视化编辑网站布局

本文档同时约束后续的模块拆分、数据边界、接口规划和实施分期。

---

## 2. 产品定位

系统定位：
- 面向跨境电商卖家、投放团队、独立站运营团队
- 提供从建站、装修、斗篷、商品订单管理到支付配置的统一后台
- 以“快速上线站点”和“持续运营多个站点”作为核心价值

典型客户：
- 需要批量搭建落地站、商城站、活动站的跨境卖家
- 需要针对不同国家、广告渠道、品类快速复制站点的团队
- 需要在同一后台管理多个站点、多个支付通道和订单流程的运营团队

---

## 3. 总体架构原则

### 3.1 架构原则

- 单体优先：现阶段继续采用单体应用，避免过早微服务化
- 多租户隔离：所有业务数据围绕 `tenant_id` 隔离
- 站点模板化：站点、页面、区块、主题配置采用结构化配置驱动
- 异步任务化：建站、域名检测、斗篷同步、订单同步、支付回调统一任务化
- 可扩展集成：ERP、支付、域名、斗篷策略引擎采用适配器模式
- 先后台后站点渲染优化：先把后台运营链路打通，再优化前台性能与复杂编排

### 3.2 非目标

当前阶段不追求：
- 微服务拆分
- 自研完整 ERP 替代成熟 ERP
- 自研支付清算网络
- 自研 CDN / WAF / DNS 基础设施

---

## 4. 目标系统上下文

```text
商户运营人员
   ↓
管理后台（Vue）
   ↓
统一业务服务（Spring Boot）
   ↓
--------------------------------------------------
| 建站引擎 | 页面装修引擎 | 斗篷引擎 | ERP 中台 |
| 支付中心 | 域名与SSL | 订单中心 | 任务调度 |
--------------------------------------------------
   ↓
外部系统
- WordPress Multisite / 站点运行容器
- 域名 DNS 服务商
- SSL 证书服务
- 支付服务商
- ERP / 仓储 / 物流接口
- 广告来源识别与风控组件
```

## 6. 核心模块架构

## 6.1 租户与账号中心

功能：
- 租户注册与初始化
- 主账号、员工账号、角色权限
- 套餐与资源额度
- 登录认证、会话、审计日志

关键约束：
- 所有业务表必须具备 `tenant_id`
- 员工权限至少支持“按站点 + 按模块”授权

## 6.2 一键建站引擎

功能：
- 基于模板一键创建站点
- 自动分配站点标识、子域名、初始页面、主题配置
- 初始化默认商品分类、页面骨架、支付位、追踪配置
- 返回可继续装修的站点工作台

推荐实现路线：
- 以 WordPress Multisite 作为站点运行底座
- 平台侧保存站点业务元数据
- 通过 Provisioner 创建子站、初始化主题和插件
- 页面布局采用平台配置驱动，不依赖人工进 WP 后台逐页调整

## 6.3 页面装修与布局编辑引擎

这是本次重构的核心模块之一。

能力要求：
- 页面树管理：首页、详情页、落地页、结账页、成功页
- 区块拖拽排序
- 区块属性编辑
- 模板保存与复用
- 多端预览
- 多语言字段扩展
- 页面发布与版本回滚

推荐数据模型：
- `page`：页面基础信息
- `page_layout_version`：页面版本
- `layout_block`：区块实例
- `layout_block_schema`：区块属性 JSON Schema
- `theme_config`：主题变量

## 6.4 斗篷引擎

斗篷模块用于根据访问来源、设备、地区、IP、UA、referer 等规则，将访问者分流到不同内容版本。

职责：
- 定义斗篷规则
- 接收请求上下文并计算命中结果
- 返回目标页面版本或跳转目标
- 记录命中日志与风险原因

规则输入：
- IP / ASN / 国家
- UA / 设备
- Referer
- 广告参数 UTM / Click ID
- 白名单 / 黑名单
- 首次访问 / 频次 / Cookie

规则输出：
- 正常页
- 审核页
- 跳转页
- 拒绝访问

设计要求：
- 规则计算必须低延迟
- 支持灰度发布和手动停用
- 斗篷判定日志必须可审计

## 6.5 ERP 中台

ERP 模块在当前阶段以“轻 ERP + 连接器”模式建设，不做重资产全量 ERP。

功能范围：
- 商品中心
- SKU 与库存
- 订单中心
- 发货状态
- 客户信息
- 物流单号
- 对接外部 ERP / WMS

能力边界：
- 平台自身提供基础商品、订单、库存能力
- 对复杂采购、财务总账、仓库波次等能力通过外部 ERP 承接

## 6.6 支付中心

功能：
- 支付通道配置
- 支付单创建
- 支付回调处理
- 支付状态同步
- 风控校验
- 退款与对账扩展位

适配策略：
- 定义统一 `PaymentGateway` 接口
- 每个渠道实现独立适配器
- 平台内部统一支付单状态机

核心状态：
- INIT
- PENDING
- SUCCESS
- FAILED
- REFUNDED
- CLOSED

## 6.7 域名与证书中心

功能：
- 默认子域名分配
- 自定义域名绑定
- DNS 检测
- HTTPS 证书申请与续期
- 域名状态告警

## 6.8 订单与履约中心

功能：
- 站点订单回传
- 支付状态合并
- 库存扣减
- ERP 同步
- 发货回写
- 售后与退款预留

## 6.9 任务调度中心

统一承接：
- 建站任务
- 模板初始化
- 域名验证
- SSL 签发
- 斗篷配置下发
- ERP 拉单 / 推单
- 支付回调补偿

## 7. 关键业务流程

## 7.1 一键建站流程

1. 商户选择站点模板、站点类型、地区、默认语言、结算币种
2. 系统校验套餐额度、子域名、模板合法性
3. 创建 `site` 主记录和初始化任务
4. 调用站点 Provisioner 创建底层站点
5. 初始化页面结构、默认区块、主题配置、支付占位、追踪脚本
6. 回写站点状态为 `ACTIVE` 或 `FAILED`
7. 商户进入站点编辑器继续装修

## 7.2 页面编辑与发布流程

1. 商户进入站点工作台
2. 选择页面并编辑布局
3. 系统保存草稿版本
4. 商户点击发布
5. 系统生成可渲染版本并刷新站点缓存
6. 前台新版本生效，可回滚至历史版本

## 7.3 斗篷访问流程

1. 访客请求站点页面
2. 系统采集请求上下文
3. 斗篷引擎按规则集计算命中结果
4. 命中“审核页”或“正常页”或“跳转页”
5. 记录命中日志、风险标签、来源信息

## 7.4 订单支付流程

1. 用户提交订单
2. 系统创建订单和支付单
3. 根据站点配置选择支付通道
4. 调用支付网关创建支付请求
5. 支付回调更新支付单与订单状态
6. 触发 ERP 同步和库存更新

## 7.5 ERP 同步流程

1. 商品、订单、物流数据写入平台中台
2. 根据站点绑定的 ERP 连接器投递同步任务
3. 调用 ERP API 推送或拉取数据
4. 记录同步结果与错误日志
5. 对失败任务进行补偿重试

## 8. 数据架构

## 8.1 核心表建议

基础域：
- `tenant`
- `user_account`
- `user_role`
- `subscription_plan`

站点域：
- `site`
- `site_domain`
- `site_template`
- `site_setting`
- `theme_config`

页面装修域：
- `page`
- `page_layout_version`
- `layout_block`
- `layout_publish_record`

斗篷域：
- `cloak_rule`
- `cloak_rule_version`
- `cloak_hit_log`

商品订单域：
- `product`
- `product_sku`
- `inventory`
- `customer`
- `order`
- `order_item`
- `shipment`

支付域：
- `payment_channel`
- `payment_order`
- `payment_callback_log`

ERP 域：
- `erp_connector`
- `erp_sync_task`
- `erp_sync_log`

平台能力域：
- `async_task`
- `operation_log`
- `audit_log`

## 8.2 数据边界要求

- 页面布局与主题配置必须独立于站点运行容器保存
- 斗篷规则与命中日志必须平台侧保存，不能仅保存在站点内
- 支付、订单、库存数据统一归平台主库
- 外部系统同步结果必须留痕，便于补偿和审计

## 9. 部署架构建议

## 9.1 MVP 部署

- `frontend`：Vue 管理后台
- `backend`：Spring Boot 单体服务
- `mysql`：主业务库
- `redis`：缓存与异步任务状态
- `wordpress-multisite`：站点运行底座
- `nginx`：域名路由与静态代理

## 9.2 后续扩展

当站点规模上升后，可按能力拆分：
- 页面渲染服务
- 斗篷规则判定服务
- 异步任务服务
- 支付回调服务

当前不建议优先拆分，先把数据模型和模块边界设计清楚。

## 10. 安全与合规

必须覆盖：
- JWT 鉴权与租户隔离
- 操作审计日志

- 管理后台细粒度权限控制

## 12. 当前系统重构结论

当前仓库已有能力：
- 多租户账号体系
- 站点注册与 Multisite 建站原型
- 基础管理后台

当前仓库不足：
- 缺少页面装修模型
- 缺少斗篷领域模型与规则执行链路
- 缺少商品/订单/库存/支付主模型
- 缺少围绕建站场景的工作台信息架构

因此后续重构方向应明确为：
- 保留现有认证、多租户、WordPress Provisioner 基础
- 降低“文章发布系统”在产品中的中心级别
- 新增建站、布局编辑、斗篷、ERP、支付四大主能力
